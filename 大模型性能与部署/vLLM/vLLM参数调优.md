## 二、性能调优的目的

 三大无代价优化优势

| 优化技术              | 性能提升      | 核心价值      |
| ----------------- | --------- | --------- |
| 异步调度              | 5-15%     | 减少GPU空闲时间 |
| 多进程模式             | 稳定性提升     | 完善的资源隔离   |
| 增强Chunked Prefill | 长序列处理效率提升 | 支持32K+上下文 |

| 参数                              | 说明                                               |
| ------------------------------- | ------------------------------------------------ |
| `async_scheduling=True` **必开启** | 异步调度，**必开启项，纯收益无副作用**                            |

| `gpu_memory_utilization=0.90`   | 高内存使用率，提高吞吐量，但是OOM风险增加<br>生产环境建议0.90-0.92，留出安全余量 |
| enable_chunked_prefill=True     | Chunked Prefill技术*建议：>4K上下文必需开启，短文本可关闭**         |

##### ⚖️ 吞吐量 vs 延迟：性能的天平

| 维度 | 🚀 高吞吐量配置 | ⚡ 低延迟配置 |
|:---|:---|:---|
| **收益** | • 吞吐量提升50-100%<br>• 适合批处理任务 | • P99延迟降低40-60%<br>• 适合在线服务 |
| **代价** | • 延迟大幅增加<br>• 首token时间变长 | • 吞吐量下降30-50%<br>• 资源利用率降低 |

##### 🎯 四大场景配置模板

| 场景 | 配置代码 | 适用场景 | 代价 |
|:---|:---|:---|:---|

| **1** | **盲目追求高内存使用率**   | `gpu_memory_utilization=0.99` 导致随机 OOM   | 生产环境建议 **≤0.92**，预留安全边界           |

##### ⚖️ vLLM 中 DP（数据并行）、PP（流水线并行）、TP（张量并行）的调优策略
```
模型能否放入单张 GPU？
├── 能 → 单卡推理（最简单）
└── 不能 → 单节点还是多节点？
    ├── 单节点（多卡在一台机器）
    │   ├── 追求低延迟 → TP（张量并行）
    │   └── 追求高吞吐 → DP（数据并行）
    │   └── 两者都要 → TP + DP 组合
    └── 多节点（跨机器）
        └── TP（节点内）+ PP（跨节点）组合
```    

| 维度 | TP（张量并行） | DP（数据并行） | PP（流水线并行） |
|------|--------------|--------------|---------------|
| **英文全称** | Tensor Parallelism | Data Parallelism | Pipeline Parallelism |
| **核心原理** | 将单层参数拆分到多 GPU，协作计算同一层 | 创建完整模型副本，各副本独立处理不同请求 | 将模型按层拆分，数据像流水线一样依次经过各 GPU |
| **处理对象** | **单个请求**的多层并行计算 | **不同请求**的并行处理 | **单个请求**的层间流水线传递 |
| **参数分布** | 每个 GPU 持有 1/N 的层参数 | 每个 GPU 持有**完整**模型参数 | 每个 GPU 持有部分层（流水线阶段） |
| **通信方式** | AllReduce（高频同步） | 无通信（完全独立） | P2P 点对点传输（阶段间传递） |
| **通信开销** | 高（需 NVLink 保障效率） | 零（无通信） | 中（流水线气泡影响） |
| **对延迟影响** | **降低单请求延迟**（并行计算） | 无影响（单请求仍在单卡） | **增加单请求延迟**（需串行经过所有阶段） |
| **对吞吐影响** | 提升（支持更大模型） | **线性提升**（并发处理） | 提升（填满流水线后） |
| **硬件要求** | 必须 NVLink/高速互联 | 无特殊要求 | 可跨节点，网络带宽要求中等 |
| **GPU 数量限制** | 必须整除注意力头数（2 的幂次） | 任意数量 | **任意数量**（支持 3,5,6,7...） |
| **适用场景** | 单节点大模型、追求低延迟 | 中小模型、追求高吞吐、高并发 | 跨节点部署、超大模型、非标准 GPU 数量 |
| **启动命令** | `--tensor-parallel-size N` | `--data-parallel-size N` | `--pipeline-parallel-size N` |
| **典型配置** | `--tp 4`（4 卡协作算一层） | `--dp 4`（4 个独立副本） | `--pp 4`（4 个流水线阶段） |
| **最佳实践** | TP 尽量在节点内（NVLink） | 单卡能放下模型时首选 | PP 跨节点，与 TP 组合使用 |
| **主要缺点** | 通信开销大，需 2 的幂次 | 单副本显存占用高 | 流水线气泡，延迟增加 |

| 场景                         | 推荐方案         | 命令示例                       | 原因                                  | 关键参数                       |
| -------------------------- | ------------ | -------------------------- | ----------------------------------- | -------------------------- |
| **模型 < 单卡显存**              | 单卡           | `vllm serve model`         | 避免并行开销                              |                            |
| 单卡放不下，单节点放得下, 模型太大，单卡 OOM  | TP           | `--tensor-parallel-size 4` | 低延迟，NVLink 通信快                      | --tensor-parallel-size N   |
| **追求极致吞吐，模型较小**, 要支持更多用户并发 | DP           | `--data-parallel-size 4`   | 无通信开销，4x 并发                         | --data-parallel-size N     |
| 单节点放不下                     | TP + PP      | `--tp 8 --pp 2`            | TP 在节点内，PP 跨节点                      |                            |
| **非 2 的幂次 GPU 数量**,跨机器部署   | PP           | `--tp 1 --pp 3`            | PP 支持任意数量（3,5,6,7...）,  TP=1，PP=任意数 | --pipeline-parallel-size N |
| 超大模型（如 405B）               | TP + PP + DP | `--tp 8 --pp 4 --dp 2`     | 三层并行组合                              |                            |
延迟 vs 吞吐量权衡下的决策：

| 指标                   | 优化方案 | 说明         |
| -------------------- | ---- | ---------- |
| **首 Token 延迟（TTFT）** | TP   | 并行计算减少等待   |
| **吞吐（tokens/sec）**   | DP   | 多副本并行处理    |
| **总延迟**              | TP   | 但受通信开销影响   |
| **性价比**              | DP   | 无通信开销，线性扩展 |

## 三、关键性能瓶颈预判

基于vLLM架构和医学大模型特点，重点关注以下瓶颈：

### 1. **输入侧瓶颈：长上下文处理**
- **风险点**：病例文本通常较长（2k-8k tokens），病史、检查报告、影像描述叠加
- **vLLM影响**：Prefix Caching命中率、Prompt计算阶段的KV Cache预填充耗时
- **测试重点**：TTFT随输入长度增长的曲线是否线性恶化

### 2. **输出侧瓶颈
- **风险点**：ICD编码需要生成结构化JSON/列表，输出长度不稳定（100-500 tokens）
- **vLLM影响**：Decoding阶段的Memory Bandwidth瓶颈、Batch调度效率
- **测试重点**：TPOT是否随输出长度增加而显著上升

### 3. **并发瓶颈：vLLM的Continuous Batching极限**
- **风险点**：医疗高峰期多医生同时问诊（预估20-50并发）
- **vLLM影响**：`max_num_seqs`、`max_num_batched_tokens`配置不当导致调度阻塞
- **测试重点**：并发数与延迟的拐点、是否出现请求饿死（starvation）

### 4. **资源瓶颈：显存碎片与KV Cache管理**
- **风险点**：病例长度差异大导致显存碎片，长病例占用大量KV Cache
- **vLLM影响**：`gpu_memory_utilization`设置、PagedAttention的块分配效率
- **测试重点**：长运行后显存是否缓慢增长（泄漏）、OOM错误率

## 四、性能测试场景设计

### 场景A：基线性能测试（单用户/单并发）
**目的**：建立性能基线，验证模型本身推理速度

| 测试项                            | 配置                                  | 观测指标               |
| ------------------------------ | ----------------------------------- | ------------------ |
| 短病例（500 tokens输入/100 tokens输出） | `max_tokens=512`, `temperature=0.1` | TTFT、总延迟、TPS       |
| 中病例（2k tokens输入/300 tokens输出）  | 同上                                  | TTFT、TPOT、显存占用     |
| 长病例（8k tokens输入/500 tokens输出）  | 同上                                  | TTFT是否>30s、是否触发OOM |
|                                |                                     |                    |

**预期问题**：长病例TTFT突增（Prefix计算耗时）、显存占用非线性增长

---

### 场景B：并发压力测试（梯度加压）
**目的**：找到vLLM在医学场景下的并发拐点

| 阶段 | 并发数 | 持续时间 | 负载特征 |
|-----|-------|---------|---------|
| 轻载 | 5 | 10分钟 | 均匀混合短/中/长病例 |
| 中载 | 20 | 15分钟 | 模拟上午门诊高峰 |
| 重载 | 50 | 10分钟 | 压力测试，观察降级行为 |
| 极限 | 100 | 5分钟 | 验证系统崩溃边界 |

**关键观测**：
- 延迟百分位变化（P50/P95/P99）
- 是否出现请求排队（Queue Time）
- vLLM的`iteration_stats`日志中的batch利用率

**预期问题**：
- 并发>30时TPOT显著增加（Memory带宽瓶颈）
- 长病例在高压下被饿死（调度不公平）

---

### 场景C：长稳测试（ soak test ）
**目的**：验证长时间运行下的资源稳定性和输出质量

| 配置 | 持续时间 | 负载 |
|-----|---------|------|
| 中等并发（15-20） | 4-8小时 | 循环使用真实病例数据集 |

**观测重点**：
- 显存是否有缓慢泄漏（每小时记录`nvidia-smi`）
- 输出质量是否随时间下降（随机抽样验证ICD编码准确率）
- 是否出现CUDA错误、NCCL超时等稳定性问题

---

### 场景D：边界与异常测试
**目的**：验证极端情况下的系统行为

| 测试项  | 场景描述                   | 预期行为            |
| ---- | ---------------------- | --------------- |
| 超长输入 | 10k+ tokens的复杂病历       | 优雅截断或拒绝，不OOM    |
| 超长输出 | 要求生成详细诊疗方案（2k+ tokens） | 在5分钟内完成，质量不崩    |
| 突发流量 | 1分钟内从5并发突增到50          | 自动扩缩容或排队，不丢请求   |
| 错误注入 | 发送格式错误/乱码请求            | 返回400错误，不影响其他请求 |
|      |                        |                 |

## 五、关键风险与应对策略

| 风险现象              | 根因分析                                   | 应对措施                                          |
| ----------------- | -------------------------------------- | --------------------------------------------- |
| **TTFT随输入长度指数增长** | Prefix Caching未命中，或Chunked Prefill关闭   | 开启`--enable-chunked-prefill`，预热常用病例模板         |
| **高并发下TPOT恶化**    | Memory Bandwidth饱和，Batch过大             | 降低`max_num_batched_tokens`，启用Tensor并行         |
| **显存缓慢增长最终OOM**   | KV Cache碎片，或`gpu_memory_utilization`过高 | 降至0.85，开启`--enable-prefix-caching`减少重复计算      |
| **长病例被饿死**        | vLLM调度策略偏向短请求                          | 考虑使用`--preemption-mode swap`或调整`max_num_seqs` |
