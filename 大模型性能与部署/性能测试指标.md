##### 完整的大模型推理性能指标体系，包含定义、计算方式和异常诊断：
 
##### 硬件资源预估（比如：QwQ-32B模型推理）
基于vLLM实现QwQ-32B模型推理的显存大小主要由以下两部分组成:
- 1. 模型参数占用: 参数显存=32B✖️2 Byte(FP16)≈64GB
- 2. KV Cache: 依赖上下文长度,2✖️32768(32K)✖️64(层数)✖️5120(维度)✖️2 Bytes(FP16)≈40 GB,由于QwQ-32B有8个KV头共享，显存减少了1/5,KV Cache显存占用大约在32G。


##### 最大并发量估计
 定义：在 **99% 请求 5 分钟内完成** 的 SLA 下的并发上限。
最大并发 ≈ (300s - TTFT_single) / (output_tokens × TPOT_single) × safety_factor(0.8)
[[最大并发量估计方法]]

客户端视角（总并发）          vLLM 服务端视角（GPU 并行）
┌─────────────────┐         ┌─────────────────────────┐
│ 在途请求: 100   │  ──►    │ Running:  max_num_seqs  │
│ (已发送未返回)   │         │ (正在 GPU 计算)          │
│                 │         │                         │
│                 │         │ Waiting:  队列中等待      │
│                 │         │ (显存不足或调度限制)       │
└─────────────────┘         └─────────────────────────┘
         ↑                           ↓
    客户端超时 300s              服务端调度策略
**核心矛盾**：
- `max_num_seqs` 设得太高 → GPU 利用率好，但单个请求变慢，容易超时
- `max_num_seqs` 设得太低 → 延迟好，但吞吐量差，队列堆积


## 一、延迟类指标（Latency）

| 指标              | 公式                                                                                      | 正常范围  | 异常阈值   | 异常时可能的问题                                          |
| :-------------- | :-------------------------------------------------------------------------------------- | :---- | :----- | :------------------------------------------------ |
| **TTFT**        | $$T_{\text{first\_token}} - T_{\text{request\_start}}$$                                 | < 2s  | > 30s  | **预填充阶段瓶颈**：输入过长、批处理队列积压、算力不足、Prompt Cache未命中     |
| **TPOT**        | $TPOT = \frac{T_{\text{last\_token}} - T_{\text{first\_token}}}{N_{\text{output}} - 1}$ | <50ms | >200ms | **解码阶段瓶颈**：KV Cache未优化、显存带宽饱和、序列过长导致Attention计算膨胀 |
| **E2E Latency** | $$TTFT + (N_{\text{output}} - 1) \times TPOT$$                                          | < 60s | > 300s | **综合瓶颈**：生成长度失控、级联故障、GPU掉卡、网络超时                   |

### 异常诊断流程
```
E2E高 → 拆解为 TTFT + TPOT × N
       ↓
    TTFT高？检查 Prefill 队列、输入长度、Prompt Cache命中率
    TPOT高？检查 KV Cache效率、显存带宽、是否触发Paging
    N过大？检查 生成长度限制、停止条件失效
```

## 二、吞吐类指标（Throughput & Goodput）

| 指标             | 公式                                                 | 正常范围         | 异常阈值             | 异常时可能的问题                                     |
| :------------- | :------------------------------------------------- | :----------- | :--------------- | :------------------------------------------- |
| **Throughput** | $$\frac{\sum N_{\text{output}}}{\Delta T}$$        | 硬件理论值的60-80% | 骤降30%+           | **系统性瓶颈**：GPU利用率低、批处理策略保守、请求分布不均、调度开销过大      |
| **Goodput**    | $$\frac{\text{E2E} < 300s \text{的请求数}}{\Delta T}$$ | 接近Throughput | < Throughput的50% | **长尾延迟严重**：部分请求卡住、资源争抢、优先级 inversion、OOM导致重试 |

### Goodput vs Throughput 关键差异
- **Throughput高但Goodput低** = "虚假繁荣"
  - 现象：系统显示吞吐很高，但用户投诉超时
  - 根因：部分请求E2E>300s被计入吞吐但未完成，或重试风暴
  - 案例：vLLM的Prefix Cache未命中导致个别请求TTFT暴涨

## 三、资源类指标（Resource）

| 指标 | 公式/定义 | 正常范围 | 危险阈值 | 异常时可能的问题 |
|:---|:---|:---|:---|:---|
| **GPU Utilization** | $$\frac{\text{GPU活跃时间}}{\text{总时间}} \times 100\%$$ | 70-90% | < 50% 或 > 95% | **<50%**：内存等待（TPOT高）、批太小、IO瓶颈<br>**>95%**：算力饱和，队列堆积，无法响应突发 |
| **KV Cache 占用率** | $$\frac{\text{已分配KV块}}{\text{总KV块容量}} \times 100\%$$ | < 80% | > 90% | **显存压力**：长上下文堆积、块碎片、触发驱逐导致TPOT抖动 |
| **Paging频率** | $$\frac{\text{CPU↔GPU块交换次数}}{\Delta T}$$ | < 10次/s | 持续>50次/s | **显存不足**：vLLM频繁swap，TPOT剧烈波动，最终OOM |

## 四、关联性矩阵（诊断速查）

| 现象组合 | 根因推断 | 紧急处理 |
|:---|:---|:---|
| TTFT↑ + GPU Util↓ | 批处理队列空转，调度延迟 | 调整 `max_num_seqs`，启用Chunked Prefill |
| TPOT↑ + Paging↑ | KV Cache溢出，触发swap | 降低 `max_num_batched_tokens`，启用Prefix Cache |
| Throughput↑ + Goodput↓ | 长尾请求堆积，重试风暴 | 设置动态超时，启用优先级调度 |
| TTFT正常 + TPOT抖动 | 块碎片或动态批处理干扰 | 调整 `block_size`，启用CUDA Graph |
| 全指标异常 + GPU Util↓ | GPU掉卡、驱动故障、NCCL超时 | 立即切流，检查硬件健康 |
