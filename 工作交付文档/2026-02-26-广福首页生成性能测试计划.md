# 业务背景
考虑到医学场景的长文本特性（2K-30K tokens）和 5 分钟超时阈值，这是一个典型的 **长上下文 + 高显存压力 + 延迟敏感** 场景。

---

### 一、 关注风险场景
- **超时风险**：5分钟 = 300s 是硬阈值，需找到并发与延迟的拐点

---

### 二、核心性能指标

| 指标类别    | 具体指标                         | 定义说明                                                                             | 业务意义         | 风险阈值           |
| ------- | ---------------------------- | -------------------------------------------------------------------------------- | ------------ | -------------- |
| **延迟类** | TPOT (Time Per Output Token) | $\frac{T_{\text{last\_token}} - T_{\text{first\_token}}}{N_{\text{output}} - 1}$ | 生成流畅度感知      | >200ms 卡顿感     |
|         | TTFT                         | $$T_{\text{first\_token}} - T_{\text{request\_start}}$$                          | 预填充阶段瓶颈      | > 2s           |
|         | E2E Latency                  | $$TTFT + (N_{\text{output}} - 1) \times TPOT$$                                   | 端到端总耗时       | **>300s 超时失败** |
| **吞吐类** | Throughput (tok/s)           | $$\frac{\sum N_{\text{output}}}{\Delta T}$$                                      | 系统总处理能力      | 越高越好           |
|         | Goodput (有效吞吐)               | $$\frac{\text{E2E} < 300s \text{的请求数}}{\Delta T}$$                               | 300s 内完成的请求数 | **核心优化目标**     |


客户端视角（总并发）          vLLM 服务端视角（GPU 并行）
┌─────────────────┐         ┌─────────────────────────┐
│ 在途请求: 100   │  ──►    │ Running:  max_num_seqs  │
│ (已发送未返回)   │         │ (正在 GPU 计算)          │
│                 │         │                         │
│                 │         │ Waiting:  队列中等待      │
│                 │         │ (显存不足或调度限制)       │
└─────────────────┘         └─────────────────────────┘
         ↑                           ↓
    客户端超时 300s              服务端调度策略
**核心矛盾**：
- `max_num_seqs` 设得太高 → GPU 利用率好，但单个请求变慢，容易超时
- `max_num_seqs` 设得太低 → 延迟好，但吞吐量差，队列堆积

## 三、vLLM 参数配置策略（预实验设计）

基于 2×L20 的 96GB 总显存，设计 **2 组基线配置**：
### 配置 A：延迟优先（Latency-Optimized）
```bash
# 目标：最小化单请求延迟，适合低并发场景
vllm serve QwQ-32B \
  --tensor-parallel-size 2 \
  --dtype bfloat16 \
  --max-model-len 32768 \
  --max-num-seqs 4 \              # 严格限制并发
  --max-num-batched-tokens 32768 \
  --gpu-memory-utilization 0.90 \
  --enable-chunked-prefill \      # 长文本预填充优化
  --prefill-chunk-size 2048       # 控制预填充步长
```

### 配置 B：吞吐优先（Throughput-Optimized）
```bash
# 目标：最大化系统吞吐量，适合批处理场景
vllm serve QwQ-32B \
  --tensor-parallel-size 2 \
  --dtype bfloat16 \
  --max-model-len 32768 \
  --max-num-seqs 16 \             # 提高并发上限
  --max-num-batched-tokens 65536 \ # 增大批处理
  --gpu-memory-utilization 0.95 \
  --enable-chunked-prefill \
  --num-scheduler-steps 8 \       # 调度步数优化
  --multi-step-stream-outputs     # 流式输出优化
```

## 五、关键问题解答框架

### 问题 1：多少并发会造成超时？
**测试方法**：固定 prompt 长度（如 4K），逐步提升并发，绘制 **超时率-并发数** 曲线。

**判定标准**：
- **安全区**：超时率 < 0%
- **警告区**：超时率 1%-5%（需限流）

**预期结论**（基于经验估算）：
- 2K tokens：并发 12-16 可能为拐点
- 8K tokens：并发 8-12 可能为拐点  

### 问题 2：吞吐量最大的参数设置？
**优化路径**：
1. 启用 `enable-chunked-prefill` + 较大的 `max-num-batched-tokens`
2. 设置 `num-scheduler-steps=8` 或更高
3. 使用 Continuous Batching 最大化 GPU 填充率
4. **约束**：需保证 P99 E2E Latency < 300s

### 问题 3：最大并发量？
**定义**：在 **100% 请求 5 分钟内完成** 的 SLA 下的并发上限。

**估算方法**：
```
最大并发 ≈ (300s - TTFT_single) / (output_tokens × TPOT_single) × safety_factor(0.8)
```

需通过实际测试验证，预计 **8K tokens 场景下约为 8-12 并发**。
