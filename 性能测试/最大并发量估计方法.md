### 基础概念

| 术语                          | 含义                                  | 备注说明                                         |
| --------------------------- | ----------------------------------- | -------------------------------------------- |
| **并发量（Concurrency）**        | 同时处于 GPU 上计算的请求数                    | `max_num_seqs`（vLLM 的**内部调度并发**，不是系统能承载的总并发） |
| **吞吐量（Throughput）**         | 单位时间完成的请求数                          | tok/s 或 req/s                                |
| Goodput (有效吞吐) ---真正的"最大并发" | **在保证 300s SLA 的前提下，系统能同时承载多少在途请求** |                                              |
| **在途请求（In-flight）**         | 已发送但未完成的请求总数                        | 受客户端和服务端共同限制                                 |
| TTFT_single                 | 单请求首字延迟（预填充阶段）                      |                                              |
| TPOT_single                 | 单 token 生成时间（毫秒级，如 50ms）            |                                              |

## 一、最大并发预估公式

```
最大并发 ≈ (300s - TTFT_single) / (output_tokens × TPOT_single) × 0.8
```

这个公式本质上是 **"时间预算分配"** 的思维：把 5 分钟（300s）的总超时阈值，拆解为**可并行化的时间槽数量**。

##### 1.1 分子部分：`(300s - TTFT_single)`
**关键洞察**：
- **TTFT（预填充阶段）** 是计算密集型的串行过程，必须完整做完才能开始生成 token
- 这部分时间**无法被并发优化**，是固定开销
- 剩余时间 `(300s - TTFT_single)` 才是**真正可用于生成 token 的预算**
```
时间分配示意图：
[========TTFT========][==========生成阶段==========]
0                    t1                        300s
                      ↑
                 可用预算 = 300 - t1
```

##### 1.2 分母部分：`(output_tokens × TPOT_single)`
**计算的是：单请求在生成阶段的总耗时**
```
单请求生成耗时 = 输出长度 × 每 token 时间
              = 500 tokens × 50ms/token 
              = 25,000ms = 25s
```

##### 1.3 除法的本质：时间槽的切分
```
最大并发 ≈ 可用时间预算 / 单个请求生成耗时
```
**类比理解**：就像一条流水线，每个工位（请求）需要占用流水线 25 秒。在 275 秒的总预算里，理论上可以排进多少个工位？
```
时间轴 →
[请求1: 25s][请求2: 25s][请求3: 25s]...[请求N: 25s]
└───────────────── 275s 可用预算 ─────────────────┘
```

##### 1.4 核心假设（理想条件）

| 假设             | 现实偏差                        |
| -------------- | --------------------------- |
| 请求**均匀到达**     | 实际可能有突发流量                   |
| 所有请求**同时开始生成** | 实际有排队等待                     |
| `TPOT` 恒定      | 长上下文时 KV Cache 压力导致 TPOT 上升 |
| 无**显存瓶颈**      | 高并发时 KV Cache 可能触发 paging   |
**为什么这是"乐观估计"**
这个公式假设的是 **"纯生成阶段的并发"**，忽略了 vLLM 的实际调度复杂性：
```
实际情况更复杂：
[预填充A][生成A][预填充B][生成B]...  ← 交错执行（Continuous Batching）
   ↑       ↑      ↑       ↑
  这些阶段会竞争 GPU 计算单元和显存带宽
```
**真实瓶颈往往在**：
- **预填充阶段堆积**：多个请求同时来做预填充，TTFT 会暴涨
- **KV Cache 显存不足**：导致请求排队或块交换（paging）
- **TPOT 随序列长度增加**：生成越多 token，注意力计算越慢

**长文本场景的修正因子**

| 场景              | 修正       | 原因                 |
| --------------- | -------- | ------------------ |
| Prompt 30K      | 并发 × 0.5 | KV Cache 占用翻倍，显存瓶颈 |
| 输出 >1000 tokens | 并发 × 0.7 | TPOT 随序列长度衰减       |
| 多用户混合长度         | 并发 × 0.6 | 短文本被长文本"阻塞"        |
## 二、 vLLM 的调度机制下估算并发量

```plain
客户端视角（总并发）          vLLM 服务端视角（GPU 并行）
┌─────────────────┐         ┌─────────────────────────┐
│ 在途请求: 100   │  ──►    │ Running:  max_num_seqs  │
│ (已发送未返回)   │         │ (正在 GPU 计算)          │
│                 │         │                         │
│                 │         │ Waiting:  队列中等待      │
│                 │         │ (显存不足或调度限制)       │
└─────────────────┘         └─────────────────────────┘
         ↑                           ↓
    客户端超时 300s              服务端调度策略
```
**核心矛盾**：
- `max_num_seqs` 设得太高 → GPU 利用率好，但单个请求变慢，容易超时
- `max_num_seqs` 设得太低 → 延迟好，但吞吐量差，队列堆积
##### 2.1 第一层：确定 `max_num_seqs`（GPU 并行度）
**目标**：找到在 300s 内能完成的**最大 GPU 并行数**
```plain
单请求总耗时 T_total = TTFT + T_gen
其中：
- TTFT = f(prompt_len, max_num_seqs)  ← 随并发增加而增加
- T_gen = output_tokens × TPOT        ← 也受并发影响
约束条件：T_total < 300s
```
**关键认知**：TTFT 和 TPOT 都是 `max_num_seqs` 的函数——并发越高，每个请求分到的算力越少。
**经验公式（基于 vLLM 的 Continuous Batching）**：
```plain
当 max_num_seqs = N 时：
- TTFT(N) ≈ TTFT(1) × (1 + α × (N-1))     // α: 预填充竞争系数，约 0.1-0.3
- TPOT(N) ≈ TPOT(1) × (1 + β × (N-1))     // β: 生成阶段竞争系数，约 0.05-0.15
```
```plain
实际情况（更复杂）：
         ┌─────────────────────────────────────┐
TTFT(N)  │  ╭─╮                                │
         │ ╱   ╲  ← 低并发区：接近恒定（有闲置算力）│
         │╱     ╲______________________________│
         │        ╲  ← 高并发区：线性增长（饱和）   │
         │         ╲___________________________│
         └─────────────────────────────────────┘
           1  2  4  8  16  N
真实行为是分段的：
- **低并发（N < GPU SM 利用率阈值）**：TTFT 几乎不变，α ≈ 0
- **过渡区**：开始排队，α 逐渐增大
- **高并发（GPU 100% 利用）**：线性增长，α 稳定
```

| 系数           | 范围    | 观察来源                                                                                                |
| ------------ | ----- | --------------------------------------------------------------------------------------------------- |
| α: 0.1-0.3   | 预填充竞争 | vLLM 社区 benchmark（如 [AnyScale 测试](https://www.anyscale.com/blog/continuous-batching-llm-inference)） |
| β: 0.05-0.15 | 生成竞争  | 生成阶段是 memory-bound，竞争比计算密集的预填充轻                                                                     |

##### 2.2  第二层：确定系统总并发（在途请求上限）
```plain
系统总并发（客户端）= max_num_seqs + 队列缓冲
```
##### 2.3 最终估算公式（修正版）
```plain
┌─────────────────────────────────────────────────────────────┐
│  STEP 1: 显存约束（硬上限）                                    │
│  max_num_seqs_mem = floor(GPU_memory × utilization / KV_per_seq) │
│                                                                 │
│  其中 KV_per_seq = 2 × num_layers × hidden_size × seq_len × 2B  │
│  对 QwQ-32B: ≈ 1.3MB/token × (prompt + output的长度)                 │
├─────────────────────────────────────────────────────────────┤
│  STEP 2: 时间约束（软上限）                                    │
│  求解 N 使得:                                                  │
│  TTFT(1)×(1+α(N-1)) + output×TPOT(1)×(1+β(N-1)) < 300s        │
│                                                                 │
│  通常 N_time >> N_mem，所以显存约束更紧                         │
├─────────────────────────────────────────────────────────────┤
│  STEP 3: 系统总并发（含队列）                                   │
│                                                                 │
│  保守策略（无流控）: 总并发 = max_num_seqs × 1.2               │
│  激进策略（有流控）: 总并发 = max_num_seqs × 2.0               │
│  安全策略（严格 SLA）: 总并发 = max_num_seqs                   │
└─────────────────────────────────────────────────────────────┘
```


## 三：vLLM 官方性能预测工具（推荐）
vLLM 0.4.0+ 提供了内置的 benchmark 工具，可以直接测量
**vLLM 官方 benchmark 测出的指标，如何帮你确定 SLA 时间内的总请求数**。
**输出**：直接的 TTFT, TPOT, Throughput 数据表，无需公式估算
```bash
# 官方推荐的性能测试方式
python -m vllm.benchmarks.benchmark_throughput \
  --model QwQ-32B \
  --tensor-parallel-size 2 \
  --max-num-seqs 1,2,4,8,16 \    # 扫描不同并发
  --input-len 8192 \
  --output-len 500 \
  --dtype bfloat16
```
```

参考理论说明
```plain
设：
- C = GPU 总算力（FLOPs/s）
- W_prefill = 预填充计算量 ≈ 2 × param_count × prompt_len
- W_gen = 生成计算量 ≈ 2 × param_count × output_len × steps
则：
TTFT(N) = W_prefill / (C / N) = N × W_prefill / C   （当 N > 饱和点）
TPOT(N) = W_gen_step / (C / N) = N × W_gen_step / C
```

假设测试结果如表：

| max_num_seqs | TTFT (s) | TPOT (ms) | Throughput (tok/s) | Latency (s) |
|-------------|---------|----------|-------------------|------------|
| 1 | 2.0 | 50 | 180 | 27 |
| 2 | 2.3 | 55 | 320 | 30 |
| 4 | 3.0 | 70 | 520 | 38 |
| 8 | 5.5 | 110 | 680 | 58 |

从这张表，你能算出 **SLA 时间内的总请求数**：
步骤 1：确定可行的 max_num_seqs
```
约束：Latency < 300s

查表：
- max_num_seqs=4 时，Latency=38s ✓
- max_num_seqs=8 时，Latency=58s ✓

甚至 max_num_seqs=16 可能也满足（需实测）
```
步骤 2：计算批次数量
```
batches = floor(300s / Latency)
        = floor(300 / 38) 
        = 7 批（当 max_num_seqs=4）
```
步骤 3：计算总请求数
```
total_requests = max_num_seqs × batches
               = 4 × 7
               = 28 个请求/300s
```

| 你的问题 | Benchmark 如何回答 |
|---------|------------------|
| **多少并发会超时？** | 找到 Latency 接近 300s 的 max_num_seqs 值 |
| **吞吐量最大怎么配？** | 对比不同 max_num_seqs 的 Throughput 列 |
| **延迟最低怎么配？** | 找 Latency 最小的配置（通常是 max_num_seqs=1 或 2）|
| **最大并发量多少？** | 用上述公式计算 total_requests |

```
Benchmark 假设：
- 所有请求同时到达（同步启动）
- 无队列等待
- 无客户端网络延迟

现实场景：
- 请求异步到达 → 需要排队模型
- 有网络开销 → Latency 增加 10-20%
- 有冷启动 → 首请求更慢
  
Benchmark 给出的是理论上限
实际总请求数 = Benchmark 计算值 × safety_factor(0.7-0.8)
```
实战：从你的 Benchmark 到部署决策
假设测得数据

| 配置                       | Latency | 决策                        |
| ------------------------ | ------- | ------------------------- |
| max_num_seqs=4, input=8K | 38s     | 选这个作为生产配置                 |
| max_num_seqs=8, input=8K | 58s     | 延迟增加 50%，吞吐提升 30%，权衡后可能不选 |

计算容量
```
单请求 8K → 500 tokens：
- 每 300s 可处理 7 批 × 4 = 28 个请求
- 换算：28 / 300s = 0.093 req/s = 5.6 req/min
  
如果业务峰值 10 req/min：
- 单实例不够，需要 2 实例
- 或降低精度提升吞吐
```

> **Benchmark 测的是"宽度"（max_num_seqs）和"速度"（Latency），你用它算出"深度"（batches），三者相乘得到 SLA 时间内的总容量。**

没有 Benchmark，你的容量规划是猜测；有了 Benchmark，你可以精确到 **"8K tokens 场景下，2×L20 可以稳定支持 28 个请求/5分钟"** 这样的具体数字。
